## 一、项目概述

### 1.1 项目名称
民宿预约管理系统

### 1.2 项目背景
随着旅游市场的发展，民宿行业快速增长，但传统民宿在预约管理上存在信息混乱、状态不透明、管理低效等问题。本系统旨在为民宿运营者提供一套完整的在线预约与后台管理工具，同时为旅客提供便捷的房源搜索与预订体验。

### 1.3 建设目标
- 为旅客提供民宿搜索、查看、预约、支付、评价的完整服务闭环
- 为管理员提供房源管理、订单处理、房态维护、评价管理、用户管理等后台能力
- 实现订单状态的全流程流转管理，确保房态与订单的实时联动

### 1.4 适用用户
| 用户类型 | 说明 |
|---|---|
| 普通用户（旅客） | 在用户端进行房源搜索、预约、下单、支付、评价等操作 |
| 后台管理员 | 在管理端进行房源管理、订单处理、用户管理、评价管理等操作 |

### 1.5 系统边界
**本系统包含：**
- 用户端：首页、房源列表、房源详情、预约下单、预约成功、我的订单、订单详情、发表评价、个人中心、我的收藏
- 管理端：后台登录、数据看板、房源管理、房态维护、订单管理、评价管理、用户管理、系统配置

**本系统不包含：**
- 真实支付网关对接（当前为模拟支付）
- 房东入驻/多商家管理（当前为单一管理员运营）
- 即时通讯/站内消息（"联系房东"保持占位入口，不实现实际功能）
- 退款到账的实际支付回调处理
- 短信/邮件通知（系统不发送任何通知）
- 用户端手机号注册/验证码登录

## 二、用户角色

### 2.1 普通用户（旅客）

| 属性 | 说明 |
|---|---|
| 使用目的 | 搜索并预约合适的民宿，完成入住后进行评价 |
| 可访问模块 | 首页、房源列表、房源详情、预约下单、我的订单、发表评价、个人中心、我的收藏 |
| 主要操作 | 搜索/筛选房源、收藏房源、提交预约、查看订单状态、取消订单、模拟支付、发表评价 |
| 访问控制 | 用户端需要登录后才能使用预约、收藏、评价等功能；登录方式为用户名+密码，不使用手机号/验证码 |

**备注：** 用户注册时仅需提供用户名和密码，不要求手机号。User 表中的 phone 字段为可选字段，用于订单联系。

### 2.2 后台管理员

| 属性 | 说明 |
|---|---|
| 使用目的 | 管理民宿运营的全部后台数据，包括房源、订单、评价、用户 |
| 可访问模块 | 数据看板、房源管理、房态维护、订单管理、评价管理、用户管理、系统配置 |
| 主要操作 | 登录后台、新增/编辑/上下架/删除房源、维护房态日历、确认/拒绝/完成订单、回复/隐藏/删除评价、禁用/启用用户 |
| 访问控制 | 后台所有页面需登录鉴权；未登录时强制跳转登录页 |
| 账号数量 | 单一管理员账号（admin/123456），不支持多账号，无权限分级 |

## 三、功能模块

### 3.1 用户端模块

#### 3.1.1 首页模块

| 属性 | 说明 |
|---|---|
| 模块目标 | 提供快捷搜索入口和热门内容展示，引导用户进入房源列表 |
| 使用角色 | 普通用户 |
| 主要功能点 | 快捷搜索（城市/关键词、入住/离店日期、人数）、热门城市快捷导航、推荐房源展示（最多3套上架房源） |
| 输入信息 | 关键词、城市、入住日期、离店日期、入住人数 |
| 输出结果 | 跳转至房源列表（携带搜索参数）；展示推荐房源卡片 |
| 业务规则 | 推荐房源仅展示状态为"上架"的房源；日期默认预填当前日期起2晚 |

#### 3.1.2 房源列表模块

| 属性 | 说明 |
|---|---|
| 模块目标 | 展示可筛选的全量房源列表，支持多条件组合搜索 |
| 使用角色 | 普通用户 |
| 主要功能点 | 关键词搜索（名称/城市/设施）、城市筛选、日期筛选、人数筛选、价格区间筛选、房型筛选、排序（推荐/价格升序/评分优先/销量优先）、可预约状态标识、收藏/取消收藏 |
| 输入信息 | 关键词、城市、入住/离店日期、人数、价格区间、房型、排序方式 |
| 输出结果 | 符合条件的房源列表（含可预约状态标识）；无结果时提示"暂无符合条件的房源" |
| 业务规则 | 下架房源显示"已下架"标识；日期或人数不满足时显示"所选日期/人数不可订"；只有通过可用性校验的房源才能点击"立即预约"按钮 |

**可预约校验规则：**
1. 房源状态必须为"上架"
2. 入住人数 ≤ 房源最大可住人数
3. 入住日期与离店日期有效（离店 > 入住，至少相差1晚）
4. 所选日期内无"锁定"记录（来自房态日历）
5. 所选日期内无已生效订单占用（订单状态非"已取消"/"已退款"才算占用）

#### 3.1.3 房源详情模块

| 属性 | 说明 |
|---|---|
| 模块目标 | 展示房源完整信息，提供预约发起入口 |
| 使用角色 | 普通用户 |
| 主要功能点 | 房源真实图片/封面展示、基本信息（房型/面积/可住人数/地址/房东）、配套设施标签、入住须知、用户评价列表（限显示状态）、费用预览卡片（按当前搜索日期计算）、收藏按钮、立即预约按钮 |
| 输入信息 | 房源ID |
| 输出结果 | 房源详情页；预约按钮点击后跳转预约下单页 |
| 业务规则 | 费用 = 每晚价格 × 晚数 + 清洁费 - 优惠金额（优惠金额由系统配置驱动）；仅展示 status=显示 的评价；可预约校验同列表模块 |

#### 3.1.4 预约下单模块

| 属性 | 说明 |
|---|---|
| 模块目标 | 收集用户入住信息，生成待确认订单 |
| 使用角色 | 普通用户 |
| 主要功能点 | 入住/离店日期选择、入住人数选择、联系人姓名/手机号填写、备注填写、实时费用计算、提交预约 |
| 输入信息 | 入住日期、离店日期、入住人数、联系人姓名、手机号、备注（可选） |
| 输出结果 | 订单创建成功（状态：待确认）→ 跳转预约成功页；校验失败时提示错误信息 |
| 业务规则 | 姓名不能为空；手机号必须为11位且以1开头；离店日期必须晚于入住日期；人数不能超过房源最大可住人数；提交前再次进行可预约校验 |
| 订单编号规则 | 格式：MS + yyyyMMdd + 4位序号（如 MS202606080001） |

**表单校验规则：**
- 姓名：非空
- 手机号：正则 `/^1\d{10}$/`
- 入住晚数：≥ 1
- 入住人数：≤ 房源 capacity
- 可预约状态：实时校验

## 四、业务流程

### 4.1 用户预约主流程

```
用户首页搜索（城市/日期/人数）
  → 进入房源列表，浏览/筛选房源
    → [有意向房源] 点击查看房源详情
      → 确认信息后点击"立即预约"
        → [可预约] 进入预约下单页
          → 填写联系人信息/备注，确认费用
            → 提交预约订单
              → 系统生成订单（状态：待确认）
                → 跳转预约成功页，提示"等待后台确认"
                  → 用户可在"我的订单"查看订单进展
        → [不可预约] 按钮置灰，提示不可预约原因
```

### 4.2 订单状态完整流转

```
待确认
  ├── 后台确认 → 待支付
  │     ├── 用户支付 → 已预订
  │     │     ├── 入住日期当天系统自动流转 → 入住中
  │     │     │     └── 后台标记完成 → 已完成
  │     │     │           └── 用户评价 → 已评价
  │     │     └── 后台发起退款申请 → 退款中
  │     │           ├── 后台同意退款 → 已退款
  │     │           └── 后台拒绝退款 → 已预订（回退）
  │     ├── 待支付超时（30分钟）系统自动取消 → 已取消
  │     └── 用户取消 / 后台取消 → 已取消
  └── 后台拒绝 → 已取消
     （用户"待确认/待支付/已预订"状态均可自行取消）
```

**状态对应操作方：**
| 状态 | 触发操作方 |
|---|---|
| 待确认 → 待支付 | 后台管理员（确认） |
| 待确认 → 已取消 | 后台管理员（拒绝）或 用户（取消） |
| 待支付 → 已预订 | 用户（模拟支付） |
| 待支付 → 已取消 | 用户（取消） |
| 已预订 → 入住中 | 系统自动流转（入住日期当天，定时任务触发） |
| 已预订 → 退款中 | 后台管理员（发起退款） |
| 已预订 → 已取消 | 用户（取消） |
| 入住中 → 已完成 | 后台管理员 |
| 已完成 → 已评价 | 用户（发表评价） |
| 退款中 → 已退款 | 后台管理员（同意） |
| 退款中 → 已预订 | 后台管理员（拒绝，回退） |
| 待支付 → 已取消 | 系统自动（超时30分钟未支付）或 用户（取消） |

### 4.3 评价流程

```
用户在"我的订单"中找到"已完成"订单
  → 点击"去评价"进入评价页
    → 填写各维度评分（5维1-5星）和文字内容
      → 可选择匿名/上传图片
        → 提交评价
          → 评价保存，status=显示
          → 订单状态变更为"已评价"
          → 房源平均评分重新计算
            ↓
后台管理员可查看评价
  ├── 回复评价（填写回复内容，保存）
  ├── 隐藏评价（不在用户端展示）
  └── 删除评价
```

### 4.4 房态日历联动流程

```
后台管理员进入"房态维护"
  → 选择房源，查看月历
    → 点击"可订"日期 → 变为"锁定"
    → 点击"锁定"日期 → 变回"可订"
    → 已订日期（订单占用）不可手动修改
      ↓
用户端搜索时
  → 系统校验所选日期范围内
    → 存在"锁定"日期 → 不可预约
    → 存在已生效订单占用日期 → 不可预约
    → 全部可订 → 可预约
```

### 4.5 异常流程

| 场景 | 触发条件 | 系统处理 |
|---|---|---|
| 搜索无结果 | 筛选条件组合无匹配房源 | 提示"暂无符合条件的房源，请调整筛选条件" |
| 预约校验失败 | 人数超限、日期冲突、房源下架 | Toast 提示具体原因，阻止提交 |
| 订单取消 | 用户主动取消（待确认/待支付/已预订） | 弹窗二次确认 → 订单状态变更为"已取消" |
| 待支付超时 | 订单处于"待支付"状态超过30分钟（可配置） | 系统定时任务自动将订单状态变更为"已取消" |
| 入住日期自动流转 | 订单处于"已预订"状态且到达入住日期当天 | 系统定时任务自动将订单状态变更为"入住中" |
| 退款流程 | 已预订状态发起退款 | 订单进入"退款中" → 管理员审核 → 同意则"已退款"，拒绝则回"已预订" |
| 评价内容为空 | 提交评价时内容为空 | Toast 提示"请填写评价内容" |
| 管理员未登录 | 直接访问后台页面 | 强制跳转至后台登录页 |
| 用户被禁用 | 管理员禁用用户后，该用户尝试登录 | 拒绝登录，提示账号已被禁用 |
| 删除房源后 | 房源已删除但订单/收藏引用 | 订单展示"房源已删除"；收藏记录同步清除 |

## 七、已确认决策

以下问题已由需求方确认，作为系统开发的最终依据：

| 编号 | 问题 | 确认结果 |
|---|---|---|
| Q1 | 用户端登录注册方式 | 用户端使用用户名+密码登录注册，**不需要手机号**。User 表新增 username 字段，phone 改为可选字段。 |
| Q2 | 禁用用户处理方式 | 禁用用户后**不允许其登录**，登录时提示账号已被禁用。 |
| Q3 | 管理员账号数量 | **仅保留一个管理员账号**，不支持多账号，无权限分级。 |
| Q4 | 后台登录验证码 | **不需要验证码**，仅需用户名+密码登录。 |
| Q5 | 评价图片上传 | **需要实现**，评价发表时支持多张图片上传。 |
| Q6 | 房源封面图片 | **支持真实图片上传**，House 表新增 cover_image 字段，替代原型的 emoji/theme。 |
| Q7 | 分页条数 | **每页固定 20 条**。 |
| Q8 | 优惠机制 | 优惠金额**由系统配置驱动**（默认30元），无优惠券/促销系统。 |
| Q9 | 待支付超时处理 | **超时自动取消订单**，由后端定时任务执行（默认30分钟，可配置）。 |
| Q10 | 房源平均评分计算 | **只统计"显示"状态的评价**的整体评分均值，隐藏评价不参与计算。 |

| Q12 | 数据看板数据来源 | **需要后端接口提供真实统计数据**，包括8项核心指标和近7日订单趋势。 |
| Q13 | 收藏数据存储方式 | **保存到后端数据库**，不使用 localStorage。 |
| Q14 | 入住中状态触发方式 | **系统在入住日期当天自动流转**为"入住中"，由定时任务触发，无需管理员手动操作。 |
| Q15 | 短信/邮件通知 | **系统不发送任何通知**，不接入第三方通知服务。 |
